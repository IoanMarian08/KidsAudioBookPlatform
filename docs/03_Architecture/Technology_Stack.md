# Technology Stack

Version: 1.0.0  
Status: Draft

## Purpose

This document defines the approved technology stack for KidsAudioBookPlatform. It is the default decision source for implementation. Introducing an alternative requires an Architecture Decision Record.

## Decision Principles

- Prefer mature, well-supported technologies.
- Optimize for maintainability before novelty.
- Keep the local development environment reproducible.
- Use managed services only where they provide clear operational value.
- Avoid overlapping tools that solve the same problem.
- Select technologies that support automated testing and observability.

## Backend

| Technology | Role | Decision |
|---|---|---|
| Java 21 | Backend language | LTS, strong performance, records, modern language features |
| Spring Boot | Application framework | Mature ecosystem and strong support for REST, security, data and observability |
| Spring Security | Authentication and authorization | Standard security layer for API and administrative access |
| Spring Data JPA | Relational persistence | Reduces infrastructure boilerplate while preserving SQL control |
| Hibernate | ORM implementation | Used behind repositories; complex reads may use native SQL |
| Bean Validation | Input and domain validation | Standard declarative validation with custom validators where needed |
| Flyway | Database migrations | Versioned, repeatable and auditable schema evolution |
| Maven | Build and dependency management | Consistent with the Java ecosystem and CI pipelines |

### Backend Rules

- Public APIs use DTOs; persistence entities never leave the infrastructure layer.
- Business rules belong in domain/application services, not controllers.
- Transactions must be short and explicit.
- Blocking operations are isolated from asynchronous consumers.
- New dependencies require a documented purpose and security review.

## Data and Storage

| Technology | Role | Decision |
|---|---|---|
| PostgreSQL | Primary transactional database | Strong relational integrity, indexing, JSON support and operational maturity |
| Redis | Cache, short-lived state and distributed coordination | Used only for data that can be rebuilt or has explicit expiry |
| MinIO locally / S3-compatible storage in production | Audio, illustrations and derived media | Keeps large binary objects outside PostgreSQL |
| CDN | Media delivery | Reduces backend load and improves playback latency |

### Storage Rules

- Each bounded context owns its tables.
- Foreign keys are used inside a context; cross-context coupling is minimized.
- Object storage keys are opaque and generated by the platform.
- Signed URLs are short-lived.
- Sensitive fields are encrypted or hashed according to `Security_Architecture.md`.

## Messaging and Background Work

| Technology | Role | Decision |
|---|---|---|
| RabbitMQ | Domain and integration events | Appropriate for durable queues, routing, retries and moderate platform scale |
| Transactional outbox | Reliable event publication | Prevents database/event inconsistency |
| Scheduled workers | Cleanup, reconciliation and campaign jobs | Jobs are idempotent and observable |

Kafka is not selected for the initial product because the expected event volume does not justify its operational complexity. Reconsider only when replay, very high throughput or long event retention becomes a verified requirement.

## Mobile

| Technology | Role | Decision |
|---|---|---|
| Flutter | Android and iOS application | One codebase, strong UI control and good mobile performance |
| Dart | Mobile language | Native Flutter language |
| Riverpod | State and dependency management | Testable, explicit and suitable for feature modules |
| GoRouter | Navigation | Declarative navigation and guarded routes |
| Dio | HTTP client | Interceptors, cancellation, retries and structured error handling |
| Drift or SQLite abstraction | Offline metadata and progress queue | Relational local persistence with migrations |
| Secure Storage | Tokens and sensitive local values | Uses platform-backed secure storage |
| just_audio / audio_service | Playback and background audio | Supports background playback and media controls |
| Firebase Cloud Messaging | Push notifications | Cross-platform push delivery |

## Admin Dashboard

The dashboard may use React with TypeScript, a mature router, a query/cache library, a form library and a component system. Exact framework versions are pinned at implementation time.

Rules:
- TypeScript strict mode is mandatory.
- Server state and UI state remain separate.
- Permissions are enforced by the backend; the UI only improves usability.
- Destructive actions require confirmation and audit metadata.

## API and Documentation

| Technology | Role |
|---|---|
| OpenAPI 3 | Machine-readable API contract |
| Swagger UI | Development and internal exploration |
| Markdown + Mermaid | Architecture and flow documentation |
| ADR documents | Important decisions and alternatives |

The generated OpenAPI contract must be validated in CI. API examples in documentation must match the real schemas.

## Observability

| Technology | Role |
|---|---|
| Micrometer | Application metrics |
| Prometheus | Metrics collection |
| Grafana | Dashboards and alert visualization |
| Loki | Centralized logs |
| OpenTelemetry | Distributed trace context and future tracing backend integration |

Every request, event and background job must carry a correlation identifier.

## Delivery and Infrastructure

| Technology | Role |
|---|---|
| Docker | Reproducible packaging |
| Docker Compose | Local development environment |
| GitHub Actions | CI/CD automation |
| Reverse proxy / ingress | TLS termination, routing and request limits |
| Secret manager | Production secrets; never committed to Git |

## Dependency Governance

- Versions are pinned and updated intentionally.
- Automated vulnerability scanning runs in CI.
- Unsupported libraries are replaced before they become operational risks.
- Licenses must be compatible with commercial distribution.
- A technology is not adopted only because it is popular; it must solve a documented need.
