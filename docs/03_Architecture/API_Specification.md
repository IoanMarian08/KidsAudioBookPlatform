# API Specification

Version: 1.0.0  
Status: Draft for implementation  
Owner: Platform Architecture  
Last updated: 2026-07-14

## 1. Purpose

This document defines the HTTP API contract for KidsAudioBookPlatform. It is the authoritative implementation reference for the Flutter mobile application, the administrative dashboard, backend services, automated tests, and future external integrations.

The API is designed for a child-safe mobile product in which a parent owns the account, manages child profiles, controls purchases and settings, while children consume curated audio stories through a simplified experience.

The specification establishes:

- resource naming and versioning;
- authentication and authorization rules;
- request and response conventions;
- validation and error contracts;
- pagination, filtering, sorting, and idempotency;
- APIs for accounts, profiles, catalogue, playback, downloads, subscriptions, notifications, and administration;
- media delivery rules;
- compatibility and deprecation policies;
- observability and security requirements.

The OpenAPI definition generated by the backend must remain consistent with this document. When implementation and documentation differ, the discrepancy must be resolved before release rather than silently accepted.

## 2. Scope

### 2.1 Included

The first public version covers:

- parent registration, login, verification, token refresh, logout, and password recovery;
- parent account and preference management;
- child profile creation and management;
- parental PIN verification and biometric enablement metadata;
- public and authenticated story catalogue browsing;
- stories, series, episodes, categories, collections, narrators, and age recommendations;
- favourites, listening progress, listening history, and continue-listening features;
- playback authorization and signed media URLs;
- offline download authorization and synchronization;
- ambient sound catalogue and preferences;
- subscription plans, trials, purchases, and entitlement status;
- in-app notifications and device registrations;
- contact and support requests;
- administrative content, user, subscription, campaign, and audit APIs.

### 2.2 Excluded from the first version

The following are intentionally excluded or reserved:

- public author self-service APIs;
- social features, messaging, public comments, and child-to-child interactions;
- user-generated story uploads;
- public third-party API access;
- direct payment-card processing by the platform backend;
- real-time voice interaction;
- advertising network callback contracts beyond an internal abstraction.

## 3. API Style

The platform exposes JSON-based REST APIs over HTTPS.

Base URL:

```text
https://api.example.com/api/v1
```

Local development:

```text
http://localhost:8080/api/v1
```

Administrative endpoints use the same versioned API and are grouped under:

```text
/api/v1/admin
```

The API follows these principles:

1. Resources use plural nouns.
2. HTTP methods express intent.
3. Request and response bodies use JSON unless explicitly documented otherwise.
4. Dates and timestamps use ISO 8601.
5. Resource identifiers are UUIDs represented as strings.
6. Monetary values use integer minor units plus ISO currency codes.
7. Breaking changes require a new major API version.
8. Commands that are not naturally CRUD operations use explicit action subresources.
9. Sensitive business rules are enforced server-side even when also implemented in the client.
10. Child-facing responses expose only information needed by the child experience.

## 4. Transport and Headers

All production traffic must use HTTPS. Plain HTTP requests must be rejected or redirected at the edge and never accepted by internal services as trusted traffic.

### 4.1 Standard request headers

| Header | Required | Description |
|---|---:|---|
| `Authorization` | Conditional | Bearer access token for protected endpoints. |
| `Content-Type` | For bodies | Must be `application/json` unless uploading media. |
| `Accept` | Recommended | `application/json`. |
| `Accept-Language` | Recommended | Preferred locale such as `ro-RO` or `en-US`. |
| `X-Correlation-Id` | Optional | Client-generated UUID for troubleshooting. Server generates one when absent. |
| `X-Device-Id` | Mobile requests | Stable app-installation identifier, not an advertising identifier. |
| `Idempotency-Key` | Selected commands | Unique key for retry-safe write operations. |
| `If-None-Match` | Optional | Enables conditional GET requests. |
| `If-Match` | Selected updates | Enables optimistic concurrency where documented. |

### 4.2 Standard response headers

| Header | Description |
|---|---|
| `X-Correlation-Id` | Correlation identifier propagated through logs and events. |
| `ETag` | Resource or representation version when caching is supported. |
| `Cache-Control` | Explicit caching policy. |
| `Location` | URI of a newly created resource. |
| `Retry-After` | Delay before retrying a rate-limited or temporarily unavailable operation. |
| `Deprecation` | Indicates a deprecated endpoint. |
| `Sunset` | Planned endpoint removal date. |

The backend must not expose framework, database, infrastructure, or stack-trace information in headers or bodies.

## 5. Authentication and Session Model

### 5.1 Tokens

The platform uses short-lived JWT access tokens and rotating refresh tokens.

Recommended defaults:

- access token lifetime: 15 minutes;
- refresh token lifetime: 30 days;
- refresh token rotation on every use;
- revocation on logout, password change, suspicious activity, or account disablement;
- device-scoped sessions;
- refresh tokens stored only as secure hashes server-side.

Access tokens include the minimum required claims:

```json
{
  "sub": "f4db7bc7-9613-4d26-aa22-541520544f67",
  "session_id": "ea3975ff-719b-4c8c-bc73-7388a992db58",
  "roles": ["PARENT"],
  "entitlements_version": 8,
  "iat": 1784058000,
  "exp": 1784058900,
  "iss": "kids-audio-book-platform",
  "aud": "kids-audio-book-mobile"
}
```

Child profiles do not authenticate independently. The parent session selects a child profile context for child-facing operations. The backend validates that the selected profile belongs to the authenticated parent account.

### 5.2 Roles

Initial roles:

- `PARENT` — owns an account and child profiles;
- `SUPPORT_AGENT` — limited support access;
- `CONTENT_EDITOR` — manages catalogue content;
- `CONTENT_REVIEWER` — reviews and approves content;
- `ADMIN` — broad operational administration;
- `SUPER_ADMIN` — exceptional privileged configuration access.

Administrative endpoints require explicit role checks. Hiding controls in the dashboard is not authorization.

### 5.3 Parent-zone authorization

Some actions require a recently verified parent-zone challenge in addition to a valid access token. Examples include:

- changing the parental PIN;
- deleting a child profile;
- modifying subscription settings;
- viewing sensitive account details;
- changing age restrictions;
- initiating account deletion.

The successful PIN challenge issues a short-lived parent-zone proof bound to the account, session, and device.

Header:

```text
X-Parent-Zone-Proof: <opaque-proof>
```

Recommended proof lifetime: 5 minutes.

## 6. Authorization Model

Every protected request must validate:

1. the access token;
2. session status;
3. account status;
4. role requirements;
5. resource ownership;
6. subscription entitlements where applicable;
7. child profile association;
8. parent-zone proof for sensitive actions;
9. content availability and regional rules;
10. request-specific business constraints.

Resource ownership must be determined from authenticated server-side identity, never from a trusted account identifier supplied by the client.

## 7. JSON Conventions

### 7.1 Naming

JSON properties use `camelCase`.

Correct:

```json
{
  "displayName": "Mara",
  "recommendedAgeMin": 3,
  "recommendedAgeMax": 5
}
```

### 7.2 Null and absent values

- Optional response properties may be omitted when not applicable.
- Clients must not rely on optional properties being present.
- `null` should be used only when absence and explicit unknown have different business meaning.
- Patch operations must clearly distinguish omitted values from explicit clearing.

### 7.3 Timestamps and dates

Timestamp:

```text
2026-07-14T18:25:43.511Z
```

Date:

```text
2026-07-14
```

All server timestamps are returned in UTC. User timezone is stored separately as an IANA timezone identifier.

### 7.4 Identifiers

Identifiers use UUIDs:

```json
{
  "id": "69bb374f-97e8-4d37-9f26-314da1de3a89"
}
```

Sequential database IDs must never be exposed publicly.

### 7.5 Money

```json
{
  "amountMinor": 2999,
  "currency": "RON"
}
```

Floating-point money values are prohibited.

## 8. Response Envelopes

Single-resource endpoints return the resource directly:

```json
{
  "id": "69bb374f-97e8-4d37-9f26-314da1de3a89",
  "title": "The Moonlight Rabbit"
}
```

Collection endpoints return an items envelope and pagination metadata:

```json
{
  "items": [],
  "page": {
    "number": 0,
    "size": 20,
    "totalElements": 0,
    "totalPages": 0,
    "hasNext": false
  }
}
```

Cursor-based feeds use:

```json
{
  "items": [],
  "nextCursor": null,
  "hasNext": false
}
```

## 9. Error Contract

Errors follow a stable Problem Details-inspired structure.

```json
{
  "type": "https://api.example.com/problems/validation-error",
  "title": "Request validation failed",
  "status": 400,
  "code": "VALIDATION_ERROR",
  "detail": "One or more request fields are invalid.",
  "instance": "/api/v1/child-profiles",
  "correlationId": "87e83f1f-4e73-49bd-99b4-9f21ed0e78fa",
  "timestamp": "2026-07-14T18:25:43.511Z",
  "fieldErrors": [
    {
      "field": "displayName",
      "code": "SIZE_OUT_OF_RANGE",
      "message": "Display name must contain between 1 and 40 characters."
    }
  ]
}
```

### 9.1 Standard error codes

| HTTP | Code | Meaning |
|---:|---|---|
| 400 | `VALIDATION_ERROR` | Invalid request fields or malformed input. |
| 400 | `BUSINESS_RULE_VIOLATION` | Request is syntactically valid but violates a domain rule. |
| 401 | `AUTHENTICATION_REQUIRED` | Missing or invalid authentication. |
| 401 | `SESSION_EXPIRED` | Session is no longer valid. |
| 403 | `ACCESS_DENIED` | Authenticated caller lacks permission. |
| 403 | `PARENT_ZONE_REQUIRED` | Recent parental verification is required. |
| 403 | `ENTITLEMENT_REQUIRED` | Feature requires an active entitlement. |
| 404 | `RESOURCE_NOT_FOUND` | Resource does not exist or is not visible to the caller. |
| 409 | `RESOURCE_CONFLICT` | Conflicting state or duplicate operation. |
| 409 | `VERSION_CONFLICT` | Optimistic locking conflict. |
| 410 | `RESOURCE_GONE` | Resource was intentionally removed. |
| 413 | `PAYLOAD_TOO_LARGE` | Upload or request exceeds limits. |
| 415 | `UNSUPPORTED_MEDIA_TYPE` | Unsupported request content type. |
| 422 | `CONTENT_NOT_PROCESSABLE` | Valid format but content cannot be processed. |
| 429 | `RATE_LIMIT_EXCEEDED` | Request rate exceeded. |
| 500 | `INTERNAL_ERROR` | Unexpected internal error. |
| 503 | `SERVICE_UNAVAILABLE` | Temporary dependency or service outage. |

Authentication and ownership failures should avoid revealing whether another user's resource exists.

## 10. Pagination, Filtering, and Sorting

Default page size: 20. Maximum page size: 100.

```text
GET /stories?page=0&size=20&sort=publishedAt,desc
```

Supported conventions:

- repeated query parameters for multi-value filters;
- comma-separated sort instructions only when explicitly implemented;
- documented allow-lists for sortable fields;
- server-enforced maximum page sizes;
- cursor pagination for high-volume timelines and notifications.

Invalid sort fields return `400 VALIDATION_ERROR` rather than being silently ignored.

## 11. Idempotency

The `Idempotency-Key` header is required for operations that may be retried and could otherwise produce duplicate effects, including:

- initiating purchase verification;
- applying store purchase events;
- creating download grants;
- submitting contact requests;
- administrative publication commands;
- bulk notification campaign execution.

The server stores the key, request fingerprint, result, and expiration. Reusing a key with different request content returns `409 IDEMPOTENCY_KEY_REUSED`.

## 12. Authentication API

### 12.1 Register parent account

```text
POST /auth/register
```

Request:

```json
{
  "email": "parent@example.com",
  "password": "correct horse battery staple",
  "locale": "ro-RO",
  "timezone": "Europe/Bucharest",
  "acceptedTermsVersion": "2026-06",
  "acceptedPrivacyVersion": "2026-06"
}
```

Response: `201 Created`

```json
{
  "accountId": "f4db7bc7-9613-4d26-aa22-541520544f67",
  "email": "parent@example.com",
  "emailVerificationRequired": true
}
```

Validation:

- normalized valid email;
- password policy enforced server-side;
- current mandatory legal versions accepted;
- rate limiting by IP, device, and email fingerprint;
- bot protection may be required based on risk.

### 12.2 Login

```text
POST /auth/login
```

Request:

```json
{
  "email": "parent@example.com",
  "password": "correct horse battery staple",
  "device": {
    "deviceId": "46976852-ee26-4ea0-8723-87ee4312390d",
    "platform": "ANDROID",
    "appVersion": "1.0.0",
    "osVersion": "15",
    "deviceName": "Pixel"
  }
}
```

Response: `200 OK`

```json
{
  "accessToken": "<jwt>",
  "accessTokenExpiresInSeconds": 900,
  "refreshToken": "<opaque-refresh-token>",
  "refreshTokenExpiresAt": "2026-08-13T18:25:43Z",
  "account": {
    "id": "f4db7bc7-9613-4d26-aa22-541520544f67",
    "email": "parent@example.com",
    "status": "ACTIVE"
  }
}
```

### 12.3 Refresh session

```text
POST /auth/token/refresh
```

Request:

```json
{
  "refreshToken": "<opaque-refresh-token>",
  "deviceId": "46976852-ee26-4ea0-8723-87ee4312390d"
}
```

The response rotates both tokens. Refresh token reuse after rotation is treated as a possible credential compromise and may revoke the session family.

### 12.4 Logout

```text
POST /auth/logout
```

Request:

```json
{
  "refreshToken": "<opaque-refresh-token>"
}
```

Response: `204 No Content`.

### 12.5 Verify email

```text
POST /auth/email/verify
```

```json
{
  "token": "<verification-token>"
}
```

### 12.6 Resend verification

```text
POST /auth/email/verification-requests
```

```json
{
  "email": "parent@example.com"
}
```

Always return a neutral response to prevent account enumeration.

### 12.7 Password recovery

```text
POST /auth/password-reset-requests
POST /auth/password-resets
```

Reset request:

```json
{
  "email": "parent@example.com"
}
```

Reset completion:

```json
{
  "token": "<reset-token>",
  "newPassword": "new correct horse battery staple"
}
```

Successful password reset invalidates existing refresh tokens.



```text

```

```json
{
```

Response:

```json
{
  "proof": "<opaque-parent-zone-proof>",
```

```text

```

## 17. Favourites and Library API

```text
GET /child-profiles/{profileId}/favourites
PUT /child-profiles/{profileId}/favourites/{storyId}
DELETE /child-profiles/{profileId}/favourites/{storyId}
```


Request:

```json
{
  "storyId": "f3986724-e79f-428c-bd5f-8378db153298",
  "episodeId": null,
  "clientSessionId": "8e89234c-2d04-4058-9393-48be1d942f78",
  "resumeFromSeconds": 128,
  "deviceId": "46976852-ee26-4ea0-8723-87ee4312390d"
}
```

Response:

```json
{
  "playbackSessionId": "c32349fd-24f8-41eb-9b52-a02eaf74cc1e",
  "manifest": {
    "type": "HLS",
    "url": "https://cdn.example.com/signed/manifest.m3u8",
    "expiresAt": "2026-07-14T19:25:43Z"
  },
  "transcript": {
    "url": "https://cdn.example.com/signed/transcript.json",
    "format": "TIMED_JSON",
    "expiresAt": "2026-07-14T19:25:43Z"
  },
  "illustrations": [],
  "progress": {
    "positionSeconds": 128,
    "completed": false
  }
}
```

The server verifies entitlement before issuing signed media access.

### 18.2 Report progress

```text
PUT /playback-sessions/{playbackSessionId}/progress
```

```json
{
  "positionSeconds": 243,
  "durationSeconds": 712,
  "state": "PLAYING",
  "sequence": 14,
  "occurredAt": "2026-07-14T18:28:12Z"
}
```

Rules:

- progress updates are idempotent by session and sequence;
- stale sequence values are ignored or rejected consistently;
- position is bounded by content duration;
- completion is determined server-side using a configurable threshold;
- excessive update frequency is rate-limited;
- the endpoint should tolerate mobile retries and temporary offline buffering.

### 18.3 Complete or end session

```text
POST /playback-sessions/{playbackSessionId}/completion
POST /playback-sessions/{playbackSessionId}/end
```

Completion is explicit but verified against progress. Ending captures stop reason and final position.

### 18.4 Listening history

```text
GET /child-profiles/{profileId}/listening-history
DELETE /child-profiles/{profileId}/listening-history
```

Clearing history requires parent-zone proof.

### 18.5 Continue listening

```text
GET /child-profiles/{profileId}/continue-listening
```

Completed stories and items below the minimum meaningful-progress threshold are excluded.

## 34. Rate Limiting

Rate limits are applied by endpoint risk and caller identity.

Examples:

| Endpoint group | Example policy |
|---|---|
| Login | Strict per IP, device, and account fingerprint. |
| PIN verification | Very strict with progressive delays. |
| Registration and reset | Strict plus bot protection. |
| Catalogue reads | Generous per token/device with CDN caching. |
| Progress updates | Frequency-bounded per playback session. |
| Search | Bounded per profile/device. |
| Admin writes | Bounded per user and action. |
| Upload intents | Strict per administrator and tenant. |

A rate-limited response uses `429` and includes `Retry-After` where practical.

## 35. Caching

Public catalogue metadata may use CDN and client caching. Personalized resources must use private caching or no-store policies.

Examples:

- categories: public cache, long max-age with ETag;
- published story details: public cache with revalidation;
- child home: private, short-lived cache;
- account and subscription: private, no shared caching;
- signed media grants: no-store;
- administrative responses: private, generally no-store.

Revocation-sensitive responses must never rely on long uncontrolled client caches.

## 36. Optimistic Concurrency

Administrative content resources expose an ETag or numeric version.

Request:

```text
If-Match: "12"
```

A stale update returns:

```text
409 VERSION_CONFLICT
```

The response may include the current version but must not silently overwrite newer changes.

## 37. File and Upload Validation

Upload rules include:

- allow-listed MIME types and extensions;
- file signature inspection;
- maximum size per purpose;
- cryptographic checksum verification;
- malware scanning;
- decompression bomb protection;
- image dimension limits;
- audio/video probing;
- metadata stripping where appropriate;
- object quarantine before approval;
- server-generated storage keys;
- no execution permissions.

The original filename is metadata only and never controls the storage path.

## 38. Localization

The platform supports localized content and interface metadata.

Rules:

- `Accept-Language` expresses preference, not authorization;
- content language is explicit on story resources;
- locale fallback order is documented and deterministic;
- missing mandatory localization blocks publication when required;
- administrative endpoints expose localization maps;
- child-facing endpoints return a resolved localization rather than unnecessary translation maps.

## 39. Privacy and Child Safety

API design must minimize collection and exposure of child data.

Requirements:

- no child email, phone number, precise location, advertising identifier, or public username;
- birth year or age band preferred over full birth date unless a justified requirement exists;
- profile display names are private to the account;
- no public profile endpoints;
- no child-to-child discovery;
- no behavioural advertising based on child activity;
- listening analytics use internal identifiers and controlled retention;
- administrative access to child data is role-limited and audited;
- exports and deletions follow account-level privacy workflows;
- logs must not contain story transcripts associated with a named child profile or sensitive free text.

## 40. Security Requirements

All endpoints must follow the Security Architecture document. Minimum requirements include:

- TLS in transit;
- strict token validation;
- secure password hashing;
- refresh token rotation;
- server-side authorization;
- input validation and output encoding;
- SQL injection prevention through parameterized persistence;
- CSRF protection where browser cookie authentication is used;
- secure CORS allow-list for the admin dashboard;
- replay protection for signed callbacks;
- secret management outside source control;
- audit trails for privileged actions;
- dependency and container scanning;
- safe error messages;
- abuse detection and rate limiting.

## 41. Logging and Observability

Every request must produce structured observability data containing, where available:

- correlation ID;
- trace ID and span ID;
- route template, not raw sensitive URI values;
- HTTP method;
- status code;
- duration;
- authenticated actor ID in controlled form;
- device/platform metadata where relevant;
- service and version;
- outcome category.

Prohibited log content includes passwords, PINs, access tokens, refresh tokens, purchase tokens, push tokens, signed media URLs, reset tokens, verification tokens, raw child free text, and secrets.

Metrics should cover traffic, latency, error rates, authentication failures, entitlement checks, playback grant failures, store verification results, media processing, and notification delivery.

## 42. API Versioning

The URI includes the major version:

```text
/api/v1
```

Within a major version, the platform may:

- add optional response properties;
- add new endpoints;
- add new optional request properties;
- add enum values only when clients are designed for unknown values;
- relax validation safely;
- add pagination metadata.

A new major version is required for:

- removing or renaming fields;
- changing field meaning or type;
- making optional fields mandatory;
- changing authentication semantics incompatibly;
- removing endpoint behaviour relied upon by supported clients.

Mobile clients must handle unknown response fields and unknown non-critical enum values safely.

## 43. Deprecation Policy

Deprecated endpoints include response headers and OpenAPI annotations.

The deprecation process includes:

1. mark as deprecated;
2. publish replacement guidance;
3. instrument usage by client version;
4. announce sunset date;
5. block unsupported application versions only after migration options exist;
6. remove in a later major version or after the documented compatibility period.

Emergency removal is allowed only for severe security, legal, or child-safety reasons.

## 44. OpenAPI Requirements

The backend must publish an OpenAPI 3.1-compatible specification for supported endpoints.

Each operation must define:

- stable `operationId`;
- summary and description;
- required roles;
- request schema;
- response schemas;
- standard errors;
- validation constraints;
- examples;
- pagination parameters;
- security scheme;
- deprecation status;
- idempotency requirement where applicable.

Generated client SDKs may be used, but generated types must not leak persistence entities or internal service models.

## 47. Decision Summary

The API contract adopts the following foundational decisions:

- parent-owned accounts with subordinate private child profiles;
- short-lived JWT access tokens and rotating device-scoped refresh tokens;
- explicit parent-zone proof for sensitive operations;
- REST/JSON over HTTPS with UUID identifiers;
- stable problem-detail errors and correlation IDs;
- signed, expiring media access rather than direct public storage links;
- server-calculated entitlements based on independently verified store purchases;
- idempotent mobile write operations and offline-friendly synchronization;
- strict separation between public child-facing resources and privileged administration;
- no public social or child-discovery APIs;
- OpenAPI as an executable contract, supported by automated compatibility tests.

This document will evolve as implementation details are validated, but changes must preserve these safety, ownership, compatibility, and operational principles.
